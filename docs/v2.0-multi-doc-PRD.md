# LawFlow v2.0 — 多文书支持 PRD

> v1.3.0 现状：仅支持民事上诉状，单流程固定走法
> v2.0 目标：从"上诉状生成器"变成"诉讼文书引擎"

---

## 1. 现状梳理

### 1.1 当前架构

```
首页(上传) → flow(OCR+AI分析) → result(生成诉状)
                                      ↓
                               流式输出(SSE)
                                复制到剪贴板
                                 保存历史记录
```

**后端** (`server.py`, 1179 行):
- `_ocr()` → OCR 识别（长图合并优化）
- `_analyze()` → AI 提取 9 个基础字段
- `_generate_stream()` → 流式生成上诉状（固定 prompt）
- `_generate()` → 非流式生成上诉状
- History CRUD: `/save-history`, `/get-history`, `/delete-history`, `/clear-history`

**前端**:
- `app/page.tsx` → 上传页，跳转到 `/flow`
- `app/flow/page.tsx` → OCR + AI 分析，跳转到 `/result`
- `app/result/page.tsx` → 显示生成结果（流式）
- `app/history/page.tsx` → 历史记录列表
- History 字段: `id, fileName, uploadTime, 案号, 案由, 原告, 被告, 判决法院, 上诉法院, 判决日期, analyzeInfo, appealText`

### 1.2 已知技术约束

| 项目 | 约束 |
|------|------|
| 后端 | Python 单文件 server.py，ThreadingTCPServer |
| AI | 火山引擎 Doubao Seed，128K context |
| 前端 | Next.js 14 App Router，生产模式 `next start` |
| 存储 | history.json 最多 20 条，uploads 最多 20 文件 |
| 部署 | PM2（后端）+ nohup（前端） |

## 2. 文书类型规划

### 2.1 第一期（v2.0）—— 6 种高频文书

| # | 文书类型 | 核心场景 | 复用现有数据 | 需要补充的字段 |
|---|---------|---------|-------------|---------------|
| 1 | **民事上诉状** | 不服一审判决，已有 | 全部复用 | 无 |
| 2 | **民事起诉状** | 新案件立案 | 原被告信息 | 诉讼请求、事实与理由 |
| 3 | **民事答辩状** | 被诉后答辩 | 答辩人信息、案号 | 答辩意见、针对原告诉求的回应 |
| 4 | **代理词** | 庭审总结陈词 | 法院信息、当事人 | 委托事项、争议焦点、争议分析 |
| 5 | **执行申请书** | 判决生效申请执行 | 判决书原文、当事人 | 执行标的、已生效判决信息 |
| 6 | **保全申请书** | 诉讼前/中财产保全 | 被申请人信息 | 保全标的、担保方式、紧急程度 |

### 3.2 Prompt 模板系统

**目录结构**:
```
/opt/lawflow/prompts/
├── appeal.json          # 民事上诉状（从现有 prompt 迁移）
├── complaint.json       # 民事起诉状
├── defense.json         # 民事答辩状
├── representation.json  # 代理词
├── execution.json       # 执行申请书
└── preservation.json    # 保全申请书
```

**模板格式**:
```json
{
  "doc_type": "民事上诉状",
  "display_name": "民事上诉状",
  "description": "不服一审判决时使用",
  "required_fields": ["案号", "上诉人", "被上诉人", "判决法院", "上诉法院"],
  "system_prompt": "你是安徽国恒律师事务所的资深诉讼律师...",
  "user_prompt_template": "请根据以下判决书信息撰写民事上诉状...\n\n## 案件信息\n{info}\n\n## 判决书原文\n{ocr_text}\n\n## 格式要求\n{format_rules}\n\n禁止事项:\n{prohibited_items}",
  "format_rules": ["标题必须为'民事上诉状'", "必须列明上诉人/被上诉人基本信息", ...],
  "prohibited_items": ["禁止Markdown符号", "禁止前言/后缀", "禁止emoji", ...]
}
```

**后端加载逻辑**:
```python
def _get_prompt(doc_type, info, ocr_text):
    """根据文书类型加载 prompt 模板，填充变量"""
    prompt_file = f"/opt/lawflow/prompts/{doc_type}.json"
    template = _load_json(prompt_file)
    
    # 提取文书专属字段
    user_vars = {
        "info": info,
        "ocr_text": ocr_text,
        "format_rules": "\n".join(f"- {r}" for r in template["format_rules"]),
        "prohibited_items": "\n".join(f"- {p}" for p in template["prohibited_items"])
    }
    
    # 渲染 template
    prompt_text = self._render_template(template["user_prompt_template"], user_vars)
    return template["system_prompt"], prompt_text
```

## 4. 前端交互设计

### 4.1 完整用户流程

```
1. 首页：上传 PDF/图片
   ↓
2. Flow 页：OCR + AI 分析（自动进行，不可中断）
   → 提取 9 个基础字段
   → 显示可编辑的信息卡片
   ↓
3. 新增 → 文书类型选择页
   → 6 个卡片式选项
   → 选中后进入生成页
   ↓
4. Result 页：流式生成 + 可编辑信息卡片
   → 顶部显示当前文书标题（如"民事上诉状"）
   → 中间为生成结果（流式打字机效果）
   → 底部可切换其他文书类型
   → 可编辑信息并重新生成
```

## 5. 数据库/数据模型

### 5.1 History 模型改动

**旧结构**:
```json
{
  "id": "file_xxx",
  "fileName": "xxx.pdf",
  "uploadTime": "...",
  "appealText": "..."
}
```

**新结构**:
```json
{
  "id": "file_xxx",
  "fileName": "xxx.pdf",
  "ocrText": "..."（存储 OCR 原文，供多文书复用）
  "uploadTime": "...",
  "analyzeInfo": {
    "案号": "...", "案由": "...", "原告": "...", "被告": "...",
    "判决法院": "...", "判决日期": "...", "判决结果": "...",
    "上诉期限": "...", "上诉法院": "..."
  },
  "generatedDocuments": {
    "appeal": {
      "content": "...",
      "generatedAt": "...",
      "legalBasis": ["...", "..."]
    },
    "complaint": {
      "content": "...",
      "generatedAt": "...",
      "legalBasis": [...]
    }
  }
}
```

**设计原则**:
- 一份 PDF 对应一条记录
- 同一记录可包含多种文书的生成结果
- OCR 原文存储以便复用
- 每种文书独立存储内容和生成时间

### 5.2 History API 改动

| 端点 | 改动 |
|------|------|
| `/save-history` | 新增 `doc_type` 和 `content` 参数，支持追加文书 |
| `/get-history` | 返回 `generatedDocuments` 结构 |
| `/get-document` | 新增：返回某文书的完整内容 |

## 8. 风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 某类文书的 prompt 效果不佳 | 中 | 高 | prompt 模板可独立修改，无需改代码 |
| OCR 信息不足以支撑某些文书 | 中 | 中 | 缺失字段让用户手动补充 |
| History 嵌套结构变复杂 | 低 | 低 | 保持简单，只存已生成结果 |
| 前端页面跳转链路复杂 | 低 | 中 | 设计时保持线性流程，少分支 |
